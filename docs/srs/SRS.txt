1. Executive Summary
The Patient Triage Agent is a mobile-first, agentic clinical decision-support prototype for emergency department triage. A nurse or doctor enters patient demographics, symptoms, vitals, wait time, risk factors, and short notes. The system returns a triage priority, reasoning, missing-data warnings, contradiction flags, recommended next actions, and simulated execution logs.

The project is designed for a two-member student team using Google Antigravity as the central development and orchestration platform. Antigravity must produce visible artifacts — task plans, implementation plans, screenshots, recordings, test outputs, decision traces, tool-call evidence, and recovery logs — because the challenge rewards agentic systems that move from input to insight to action, not only functional apps.

Version 2.0 restructures responsibilities so both members own equal workloads: comparable lines of code, comparable complexity, comparable demo-facing value. No phase is idle time for one member while the other works.


2. Challenge Alignment
The challenge asks for an Autonomous Content-to-Action Agent that ingests multiple content sources, extracts insights, resolves conflicting evidence, generates 3–5 interconnected actions, simulates execution, handles partial failure, and visualises outcome changes. The Patient Triage Agent maps cleanly to these requirements by treating each patient case as a multi-source operational signal.


3. Medical Safety Positioning
This section is non-negotiable. The project must be framed as a supervised decision-support prototype. It must not claim to diagnose disease, prescribe treatment, replace clinical judgment, or be safe for real patient care without formal clinical validation.


4. Product Vision and Scope
4.1 Vision Statement
TriageFlow AI helps emergency staff prioritise patients faster, more consistently, and more transparently by combining symptoms, vitals, queue pressure, and resource availability into an explainable triage action plan.
4.2 Problem Statement
In many hospitals, emergency departments face overcrowding, long waits, insufficient staffing, and manual prioritisation pressure. A patient with severe risk can wait too long if information is incomplete, contradictory, or buried in notes. The project solves this as a prototype by giving triage staff a structured, explainable, and auditable urgency assessment.

10. Triage Decision Logic
A hybrid engine is used: hard safety rules first, then scoring. Hard rules prevent a dangerous case from being downgraded by a weighted average. The weighted score differentiates non-critical cases.


10.1 Hard Red-Flag Rules
If any critical rule is true the engine returns Red or Orange regardless of weighted score. Thresholds are simplified for prototype demonstration and must be reviewed by clinicians before any real use.


10.2 Weighted Risk Score
risk_score = 0.35 * vitals_risk + 0.25 * symptom_risk + 0.15 * vulnerable_patient_risk + 0.10 * pain_duration_risk + 0.10 * wait_time_risk + 0.05 * resource_pressure_risk

confidence = 1.0 - missing_data_penalty - contradiction_penalty - stale_data_penalty

11. Agentic Workflow and Action Chain
The system is demonstrated as an agentic workflow, not a single prediction. The agent must analyse, decide, act, verify, recover, and update state.
